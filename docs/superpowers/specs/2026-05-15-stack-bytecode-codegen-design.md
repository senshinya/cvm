# Stack Bytecode Codegen Design

Date: 2026-05-15
Status: Pending written-spec review

## Context

The current cvm pipeline is:

```text
preprocessor -> parser -> sema.Analyze -> *sema.Program
```

`compiler.go` currently stops after semantic analysis, except for `--dump-ir`.
The sema layer already provides the main codegen contract: typed IR nodes,
function/global symbols, `GlobalID`, `SlotID`, object layout metadata, labels,
and `ValidateProgramInvariants`.

This design adds the next compiler stage: generation of a stack-machine
bytecode module. The generated bytecode should be JVM-like in its execution
model, while still representing C99's address, object, data segment, and layout
semantics precisely.

The interpreter is not part of this stage. The bytecode model and codegen are
part of this stage and are designed to be complete for the current C99 sema IR.

## Goals

- Add a structured bytecode IR that can be executed by a future stack VM.
- Add codegen from `*sema.Program` to `*bytecode.Module`.
- Add `--dump-bytecode` with stable printer output for golden tests.
- Keep sema as a frontend package; bytecode and codegen live behind separate
  package boundaries.
- Model complete C99 backend semantics, including globals, static locals,
  strings, arrays, structs, unions, bit-fields, pointers, function pointers,
  aggregate initialization, VLA, volatile access, varargs metadata, switch,
  labels, and goto.
- Make generated bytecode verifiable: label references, stack discipline,
  local slots, object layouts, globals, constants, and control flow must be
  checked after generation.

## Non-Goals

- Implementing the bytecode interpreter.
- Replacing the sema IR printer.
- Performing optimization passes beyond required lowering normalization.
- Matching a native platform ABI. cvm bytecode has its own VM calling
  convention, while carrying target layout metadata for C object semantics.

## Architecture

Add two packages:

```text
bytecode/   executable bytecode model, module metadata, printer, validator
codegen/    lowering from sema.Program to bytecode.Module
```

The compiler pipeline becomes:

```text
preprocessor
  -> parser
  -> sema.Analyze
  -> sema.ValidateProgramInvariants
  -> codegen.Generate
  -> bytecode.ValidateModule
  -> bytecode.PrintModule
```

`compiler.go` gains `DumpBytecode bool`. `main.go` accepts
`--dump-bytecode`. `--dump-ir` and `--dump-bytecode` are mutually exclusive so
CLI output remains stable and easy to test.

Package dependencies are one-way:

```text
compiler -> codegen -> bytecode
compiler -> sema
codegen  -> sema
bytecode -> no sema dependency
```

`bytecode` does not duplicate sema's full C type system. It stores the runtime
value types, object layouts, data segments, and symbolic references needed by
the VM and validator.

## Bytecode Model

`bytecode.Module` contains executable functions and module-level metadata:

```go
type Module struct {
    Target    TargetInfo
    Globals   []Global
    Functions []Function
    Strings   []StringConst
    Layouts   []ObjectLayout
    Sigs      []FuncSig
}
```

### TargetInfo

`TargetInfo` records the layout facts bytecode execution depends on:

- byte order
- pointer size and alignment
- scalar sizes and alignments
- bool representation
- bit-field layout policy identifier
- layout version string

The implementation uses cvm's current default layout as the first concrete
`TargetInfo`. The bytecode format keeps this explicit so additional target
configurations can be added without redesigning the IR.

### Globals

`Global` covers variables, static locals, functions, and extern declarations.

Variable-like globals carry:

- name
- storage/linkage metadata
- size and alignment
- zero-fill size
- initialization bytes
- relocations for addresses of globals, functions, and strings
- readonly flag for string/static const data when known

Function-like globals point to a `Function` index or mark an extern function
symbol. Function IDs continue to follow sema's `GlobalID` contract.

Global and static initialization is represented as data, not as startup
instructions. Address constants are expressed with relocations.

### Functions

`Function` contains:

- name/global ID
- signature ID
- parameters
- local slots
- local objects
- dynamic object records
- label table
- linear instruction list
- optional max stack depth, computed by validation or a helper

Local storage is split into three classes:

```text
local slots       scalar, pointer, and floating values accessed like JVM locals
local objects     addressable automatic objects, aggregates, arrays, compound literals
dynamic objects   VLA and other runtime-sized frame objects
```

Only scalar locals that are non-volatile, non-aggregate, not arrays, not
address-taken, and do not require object identity may use typed local slots.
All other local storage is accessed through an address path.

### Value Types

Bytecode value types describe stack values and memory load/store width:

```text
Void
Bool
I8, I16, I32, I64
U8, U16, U32, U64
F32, F64, FLong
Ptr
ObjectAddr
```

Aggregates are not ordinary stack values. Arrays, structs, unions, and compound
literals are represented by object addresses and layout metadata. Aggregate
copy, initialization, argument passing, and return use object memory
operations.

### Object Layouts

`ObjectLayout` is the backend view of C object layout:

- size and alignment
- array element size where applicable
- field offsets for structs/unions
- bit-field container type, byte offset, bit offset, width, signedness, and
  volatility/layout policy
- optional source name for printer/debugging

Codegen consumes sema/layout information and lowers it to stable layout IDs.
Instructions refer to byte offsets and layout IDs, not to sema AST nodes.

## Instruction Semantics

The bytecode is a JVM-like typed stack machine:

- Instructions consume operands from the operand stack and push results.
- Local slots are indexed and typed.
- Branch targets are labels.
- Function calls use a VM calling convention based on the operand stack.
- C object identity and aliasing are modeled through object addresses and typed
  memory operations.

The Go representation may use a compact `Opcode + ValueType` form, but the
semantic instruction and printer output remain strongly typed, e.g. `I32Add`,
`F64LoadLocal`, and `PtrStoreLocal`.

Instruction families:

```text
constants:
  I32Const, I64Const, U32Const, U64Const, F32Const, F64Const, NullPtr
  AddrString, AddrGlobal, AddrFunc, LoadConst

stack:
  Dup, Pop, Swap

locals:
  LoadLocal(type, slot)
  StoreLocal(type, slot)
  AddrLocalObject(objectID)

dynamic objects:
  AllocDynamicObject(objectID, size, align)
  FreeDynamicObject(objectID)
  DynamicObjectAddr(objectID)

memory:
  Load(type, align, volatile)
  Store(type, align, volatile)
  MemCopy(size, align, volatile)
  MemSet(size, value, align, volatile)
  Offset(bytes)
  FieldAddr(layoutID, fieldID)
  BitFieldLoad(layoutID, fieldID)
  BitFieldStore(layoutID, fieldID)

pointer:
  PtrAdd(elemSize)
  PtrDiff(elemSize)
  PtrEq, PtrNe, PtrLt, PtrLe, PtrGt, PtrGe

integer:
  I32Add, I32Sub, I32Mul, I32DivS, I32DivU, I32RemS, I32RemU
  I64Add, I64Sub, I64Mul, I64DivS, I64DivU, I64RemS, I64RemU
  I32Neg, I64Neg
  I32And, I32Or, I32Xor, I32Shl, I32ShrS, I32ShrU
  I64And, I64Or, I64Xor, I64Shl, I64ShrS, I64ShrU
  I32Eq, I32Ne, I32LtS, I32LtU, I32LeS, I32LeU, I32GtS, I32GtU, I32GeS, I32GeU
  I64Eq, I64Ne, I64LtS, I64LtU, I64LeS, I64LeU, I64GtS, I64GtU, I64GeS, I64GeU

floating:
  F32Add, F32Sub, F32Mul, F32Div, F32Neg
  F64Add, F64Sub, F64Mul, F64Div, F64Neg
  F32Eq, F32Ne, F32Lt, F32Le, F32Gt, F32Ge
  F64Eq, F64Ne, F64Lt, F64Le, F64Gt, F64Ge

casts:
  Trunc, ZExt, SExt
  FExt, FTrunc
  IntToFloat, FloatToInt
  PtrToInt, IntToPtr
  BitCast
  BoolCast

control:
  Label
  Jump
  JumpIfZero(type)
  JumpIfNonZero(type)
  Switch(type, defaultLabel, cases)
  Return(type)
  ReturnVoid
  ReturnObject(size, align)
  Unreachable

calls:
  Call(globalID, sigID, argc)
  CallIndirect(sigID, argc)
  VaStart
  VaArg(type)
  VaEnd
```

The instruction set is complete for codegen. Printer and validator support the
full model in this stage. Interpreter support is intentionally outside this
stage.

## Stack Discipline

The validator tracks stack types through the function body.

Rules:

- Function entry starts with an empty operand stack.
- Statement-level labels, loop headers, switch cases, goto targets, and jump
  targets enter with an empty operand stack.
- Expression-level joins, such as short-circuit and conditional-expression
  lowering, may declare a non-empty stack signature.
- All incoming edges to a label must agree on stack height and value types.
- Function exits must match the declared return convention.

This keeps generated bytecode verifiable without requiring a full optimizer or
SSA construction.

## Calling Convention

cvm bytecode uses a VM calling convention:

- Scalar and pointer arguments are pushed on the operand stack left to right.
- Direct calls use `Call(globalID, sigID, argc)`.
- Function pointer calls push the callee address and use `CallIndirect`.
- Scalar returns are pushed on the caller's stack.
- `void` returns push nothing.
- Aggregate arguments are passed by object address or by caller-created copy,
  according to the function signature metadata.
- Aggregate returns use a hidden caller-provided return object and
  `ReturnObject(size, align)`.
- Varargs are represented by signature metadata and `VaStart`, `VaArg`, and
  `VaEnd` bytecode operations. Their concrete storage convention belongs to
  the VM specification, not to a native platform ABI.

## Codegen Lowering

`codegen.Generate(*sema.Program)` returns `(*bytecode.Module, error)`.

Before lowering, it runs or requires:

```text
sema.ValidateProgramInvariants(prog)
```

After lowering, it runs:

```text
bytecode.ValidateModule(module)
```

Codegen is organized around:

```go
emitStmt(stmt sema.Stmt)
emitValue(expr sema.Expr)
emitAddress(expr sema.Expr)
emitInitializer(dst Address, init sema.Expr, typ sema.Type)
emitStaticInitializer(init sema.Expr, typ sema.Type) InitData
lowerType(typ sema.Type) LoweredType
lowerLayout(typ sema.Type) ObjectLayoutID
```

`emitValue` pushes an rvalue. `emitAddress` pushes an object address for an
lvalue. This split is the central rule that preserves C addressability,
aliasing, assignments, increments, member access, array indexing, dereference,
and volatile behavior.

### Declarations

Codegen covers:

- global variables
- file-scope static variables
- block-scope static variables
- extern declarations
- function declarations
- function definitions
- typedef and tag declarations as metadata only when needed for layout names

Function and variable IDs follow sema's assigned `GlobalID` and `SlotID`.

### Statements

Codegen covers:

- block
- declaration statement
- expression statement
- if/else
- while
- do/while
- for
- switch/case/default
- break
- continue
- goto
- label
- return
- empty statement

`break` and `continue` are resolved with a codegen context stack. `goto` and
labels use sema's resolved label pointers and stable bytecode labels.

### Expressions

Codegen covers:

- integer, floating, character, and string literals
- variable references
- enum references
- unary operations
- binary operations
- assignment
- compound assignment
- function calls
- function pointer calls
- member and arrow access
- array indexing
- conditional expression
- `sizeof`, including runtime VLA size
- comma expression
- implicit and explicit casts
- compound literals
- initializer lists
- address constants

For C expressions with unspecified operand order, codegen chooses a stable
left-to-right lowering order. This is a permitted implementation choice and
makes bytecode output deterministic. C sequence rules still receive explicit
lowering for short-circuit operators, conditional expressions, comma
expressions, and function calls.

### Local Slot Versus Address Path

Automatic scalar locals and parameters use typed local slots only when all are
true:

- scalar or pointer value
- not volatile
- not an array or aggregate
- address is never required by the lowered expression
- no object identity is required

All other objects use the address path:

```text
AddrLocalObject / AddrGlobal / DynamicObjectAddr
  -> Offset / FieldAddr / PtrAdd
  -> Load / Store / MemCopy / BitFieldLoad / BitFieldStore
```

### Assignments

For scalar local slots:

```text
emitValue(rhs)
StoreLocal(type, slot)
```

If the assignment expression value is needed, codegen preserves it with `Dup`
or reloads it according to the type and side-effect constraints.

For addressable objects:

```text
emitAddress(lhs)
emitValue(rhs) or emitAddress(rhs)
Store / MemCopy / BitFieldStore
```

Compound assignment evaluates the lvalue address once, loads the old value,
computes the new value, stores it back, and leaves the result value when the C
expression context requires it.

### Control Flow

Structured control flow lowers to labels and jumps. `switch` may lower to a
`Switch` instruction or to compare/jump chains, but the bytecode model supports
`Switch` directly. All statement-level targets use empty-stack labels.

Logical `&&` and `||` lower with short-circuit labels and produce a normalized
boolean/integer result.

### Initializers

Global and static initializers become module data:

- zero-fill
- fixed bytes
- string bytes
- aggregate bytes
- relocations for addresses

Automatic scalar initializers become ordinary instructions. Automatic aggregate
initializers write to local object storage with stores, `MemSet`, `MemCopy`, or
field/element stores.

### VLA

VLA bound expressions are evaluated at runtime. Codegen emits dynamic object
allocation with size and alignment, stores the dynamic object record, and
releases it when leaving the owning scope. `sizeof(VLA)` uses the recorded
runtime size instead of recomputing side effects.

### Bit-Fields

Bit-field access uses layout metadata. A load fetches the container, applies
mask/shift/sign extension, and pushes the logical value. A store loads the
container, replaces the field bits, and stores the container back. Volatile
bit-fields preserve volatile memory operation markers.

## Error Handling

Codegen errors include:

- source position
- sema node type
- lowered operation
- reason

In the final target, unsupported sema nodes are not a valid steady state.
`codegen.Generate` may return errors only when:

- sema invariants are violated
- layout information is missing or inconsistent
- sema produced insufficient information to determine bytecode behavior
- bytecode validation catches an internal codegen bug

Such errors indicate a frontend/layout/codegen contract problem to fix, not an
intentional feature gap.

## Printer

`bytecode.PrintModule` produces deterministic text:

- module target header
- globals in ID order
- strings in ID order
- layouts in ID order
- functions in global/function order
- local slots and local objects before instructions
- labels printed as `L0`, `L1`, ...
- relocations printed with symbolic target names and IDs

The format is optimized for golden tests and human inspection. It is not the
binary bytecode format.

## Validator

`bytecode.ValidateModule` checks:

- global, function, string, layout, and signature references
- duplicate IDs
- local slot type consistency
- local object and dynamic object references
- label existence
- stack height and stack value types across control-flow edges
- empty-stack rule for statement-level labels
- instruction operand validity
- return convention
- data segment relocation bounds
- layout field and bit-field bounds

Validation runs in codegen tests and compiler integration tests.

## Tests

### bytecode package

- opcode signature table covers every opcode
- validator accepts minimal valid modules
- validator rejects bad stack transitions, missing labels, bad references, and
  invalid data relocations
- printer output is stable for globals, functions, strings, layouts, labels,
  data, and relocations

### codegen package

- every `sema/testdata/pass/*.c` gains a `.bytecode.golden`
- additional backend-contract tests cover:
  - volatile scalar and aggregate access
  - bit-field read/write
  - VLA allocation and `sizeof(VLA)`
  - compound literals
  - function pointers and indirect calls
  - varargs metadata and intrinsics
  - static locals
  - string constants and data relocations
  - aggregate argument and return
  - switch, goto, labels, break, continue
  - pointer arithmetic and pointer difference
  - struct, union, array, and nested initializer lowering

### compiler integration

- `--dump-ir` remains unchanged
- `--dump-bytecode` prints stable bytecode text
- `--dump-ir --dump-bytecode` reports a clear mutual-exclusion error
- compiler errors display source positions through the existing error path

### Release checks

```text
go test ./... -count=1
go vet ./...
```

## Completion Criteria

This stage is complete when:

- `bytecode.Module`, printer, and validator exist.
- `codegen.Generate` lowers every sema IR node currently produced by successful
  C99 analysis.
- The compiler supports `--dump-bytecode`.
- Golden tests cover existing sema pass programs and added backend-contract
  cases.
- Bytecode validation runs after generation.
- The full Go test suite and vet pass.
